# Simu SMS 接码平台 — 全功能完成度分析报告

## 一、架构总览

| 层级 | 技术栈 | 说明 |
|------|--------|------|
| 前端 | Flutter (Dart) + GoRouter | iOS 为主，支持 Web 构建 |
| 后端 | PHP 原生（无框架） | 模块化路由 `/backend/routes/*.php` |
| 数据库 | MySQL (MariaDB) | `sms_receiver` 数据库 |
| 上游 SMS | HeroSMS API | 获取虚拟号码和验证码 |
| 支付 | Apple IAP (StoreKit) | 内购验证 |
| 部署 | 远程服务器 | `smsapi2.niceapp.eu.cc` |

## 二、完整用户流程分析 (Walkthrough)

### 1. 注册流程 ✅ (完成度 ~90%)

**前端** → **后端 API**
- LoginScreen → `/auth/manual-register` (POST) 或 quickRegister 快捷注册
- 支持：邮箱+密码注册 / 邮箱快速注册（随机生成密码）
- 注册成功自动颁发 Token 并跳转到首页

**问题**：
- ⚠️ 无邮箱验证，任何人用任意邮箱均可注册
- ⚠️ 快速注册密码使用 `DateTime.now().millisecondsSinceEpoch` 生成，可被预测
- ⚠️ 新用户有 5-10 积分注册奖励，容易被刷（无验证码/IP限制）

### 2. 登录流程 ✅ (完成度 ~95%)

**前端** → **后端 API**
- LoginScreen → `/auth/password-login` (POST)
- 支持 username 或 email 登录
- 服务端使用 `password_verify()` + bcrypt 验证

**问题**：
- ❌ 忘记密码 `/auth/forgot-password` **直接返回新密码明文给客户端**，无邮件验证

### 3. 充值流程 ✅ (完成度 ~80%)

**前端** → **后端 API**
- PaymentScreen → GET `/payment/packages` → 展示积分套餐
- 用户选择套餐 → Apple IAP 弹出支付 → 收据验证 `/verify-receipt`
- 首充双倍积分（服务端计算）

**关键问题**：
- ❌ **前端硬编码 API Key**: `606e960bb9bf523fd52b9df68d2ad44e549fef6b9a48ccdcbadd3ca90d5b3077` 暴露在 app_config.dart 中
- ❌ `PaymentScreen._handlePurchase()` → `paymentProvider.startApplePurchase()` → Apple IAP → `_handlePurchaseUpdated()` → `purchasePackage()` → POST `/verify-receipt`
  - 但如果 Web 端支付，receipt 数据生成伪造的 `web_purchase_{timestamp}` 收据，服务端会尝试验证 Apple 收据然后失败
- ❌ `/verify-receipt` 中如果 `productId` 不匹配任何套餐，会**回退到最便宜的套餐**（payment.php:110），可能导致用户付费拿到错误积分
- ⚠️ 仅支持 Apple IAP，无 Stripe/PayPal/USDT 等替代支付

### 4. 浏览服务/下单流程 ✅ (完成度 ~85%)

**前端** → **后端 API**
- HomeScreen → GET `/services` → 显示服务网格
- 点击服务 → `ServiceCountriesScreen` → GET `/service-countries?service_id=X` → 展示可用国家
- 点击国家 → `PurchaseConfirmScreen` → GET `/price/calculate` → 确认购买

**问题**：
- ❌ `_getDisplayPrice()` 在 `service_countries_screen.dart:73` 使用硬编码公式 `price * 100 * 4` 而不是从 API 获取价格，**与后端系数计算不同步**
  - 后端 `/service-countries/published` 使用价格系数 `coefficient_before=4` / `coefficient_after=4.5` + 会员折扣
  - 前端和后台可能显示不同价格
- ⚠️ 国家页面显示的号码是假数据（`_generateFakeNumber()`），只用于视觉预览
- ⚠️ 无库存显示 — 用户下单时可能 HeroSMS 无号码

### 5. 激活订单流程 ⚠️ (完成度 ~65%)

**前端** → **后端 API**
- OrderDetailScreen → 用户点击"激活" → POST `/orders/{id}/activate`
- 后端调用 `heroSMS->getNumber()` → HeroSMS 分配号码 → 存入数据库返回前台
- 前台显示号码 + 20分钟倒计时 + 开始轮询 SMS

**核心问题**：
- ❌ **SMS 轮询有 Bug** — `_startSmsPolling()` 3秒调用 `loadOrderDetail()` 只是从 DB 读取，**不调用 HeroSMS 查收短信**
  - 实际查 HeroSMS 短信的 API 是 `/orders/{id}/sms`，只能在手动点刷新时触发
  - **自动轮询永远不会收到短信！**
- ❌ 订单激活成功页面的 `_startSmsPolling()` -> `loadOrderDetail()` 只是读 DB，而 DB 中的 `sms_code` 字段只由 webhook 更新
  - 查了一下，**webhook.php 是否存在需要验证**，如果不存在，SMS 通知机制完全缺失
- ❌ 前端 `getSmsCode()` 在成功获取 SMS 后立即将本地订单状态改为 'completed'，但后端订单状态仍为 'active'
  - 状态不同步可能导致后续操作问题

### 6. 订单管理 ✅ (完成度 ~75%)

**前端** → **后端 API**
- OrdersScreen → GET `/orders` → 分 tab 展示（待处理/激活/已完成/过期）
- 支持取消订单（待处理状态可取消，退积分）
- 支持复制号码/SMS码

**问题**：
- ⚠️ Tab 筛选在**前端进行** (`provider.orders.where((o) => o.status == status)`)，而非后端 SQL 筛选
  - 当订单量增大后效率低下
- ⚠️ 取消订单后，如果已激活状态但未收到 SMS，无法退款（后端只允许 pending 状态取消）

### 7. 用户个人中心 ✅ (完成度 ~90%)

- ProfileScreen 显示余额、会员等级、消费进度
- 通知列表、交易记录、语言切换（中/英）
- 隐藏开发者模式（5次点击版本号进入 API 检测）

## 四、关键问题清单 (按严重度排序)

### 🔴 P0 — 阻塞级

1. **SMS 轮询机制断裂**
   - 文件：`order_detail_screen.dart:52`, `order_provider.dart:251-311`
   - `_startSmsPolling()` 调用 `loadOrderDetail()` 仅读 DB，不会触发 HeroSMS 查询
   - 用户需手动点"刷新"按钮才能检查 SMS
   - **修复**：轮询应调 `getSmsCode(orderId)` 而不是 `loadOrderDetail(orderId)`

2. **缺少 Webhook 回调处理 SMS 状态**
   - router.php 定义了 `/webhook` 路由指向 `webhook.php`
   - 该文件如果不存在或未实现，HeroSMS 的回调通知将无人处理
   - 订单状态永远停留在 "active"，不会自动变为 "completed"

3. **提权/未授权访问风险**
   - `user_id` 通过 URL 查询参数传递，理论上可篡改查看他人订单
   - 后端虽有 `requireOrderOwner()` 检查，但需要确认其实现是否正确

### 🟠 P1 — 严重

4. **忘记密码直接返回新密码明文**
   - 文件：`backend/routes/auth.php:155-161`
   - `forgot-password` API 直接返回新密码，无邮件发送验证
   - 即使有人知道邮箱，也无法阻止密码被返回（需要在生产中禁用该端点或加上邮件验证）

5. **前端硬编码 API Key**
   - 文件：`app_config.dart:11-14`
   - API Key 是 app 启动时的 "秘密"，反编译即可获取
   - 应使用动态签名或 OAuth 流程

6. **前端价格和后端系数不同步**
   - 文件：`service_countries_screen.dart:73`
   - 前端显示价格 `price * 100 * 4`，后端 `/service-countries/published` 使用 `coefficient_before/after` 和会员折扣
   - 用户看到的价格和实际扣款可能不同

7. **Web 端支付收据伪造**
   - 文件：`payment_provider.dart:119-125`
   - 非 iOS 平台生成伪造收据 `web_purchase_{timestamp}`
   - 服务端 `/verify-receipt` 调用 Apple 验证会失败

8. **已激活订单取消不退积分**
   - 文件：`orders.php:277-278`
   - 仅 pending 状态的订单可取消退款
   - 激活后未收到 SMS 的用户无法退款

### 🟡 P2 — 一般

9. **新用户注册无防刷机制** — 无验证码、无 IP 限流
10. **订单列表全部加载** — 无分页，前端筛选，订单量大时性能差
11. **`SettingsScreen` 功能未知** — 文件存在但未读取，可能功能未完成
12. **没有 SMS 模板匹配展示** — 只返回 code，没有显示原始短信全文
13. **Apple IAP 初始化可能阻塞启动** — `AppleIAPService().init()` 在 main 中 await

## 五、核心流程数据流重建

```
注册流程:
  LoginScreen/RegisterScreen
    → POST /auth/manual-register {email, password, device_id}
    → backend: 创建用户, 生成 UUID, 发放注册奖励 5-10积分
    → 返回 {user, token}
    → 前端存储 token, 跳转 /home

充值流程:
  PaymentScreen
    → GET /payment/packages → 显示套餐列表
    → 点击购买 → Apple IAP 弹窗 → 支付成功
    → POST /verify-receipt {receipt_data, transaction_id, user_id, product_id}
    → backend: 验证 Apple 收据, 检查防重放, 双倍首充, 加积分
    → 返回 new_balance

下单流程:
  HomeScreen → ServiceCountriesScreen → PurchaseConfirmScreen
    → POST /orders/create {service_id, country_id, quantity, price_points}
    → backend: 事务, 检查余额, 扣积分, 创建订单记录
    → 返回 orderId, 前端跳转订单详情

激活流程:
  OrderDetailScreen → 用户点"激活"
    → POST /orders/{id}/activate
    → backend: 调 heroSMS->getNumber(), 获取号码
    → 更新 order: status=active, phone_number, hero_order_id, expires_at=20min
    → 返回 phone_number, 前端显示号码

收码流程:
  OrderDetailScreen → 自动轮询 (当前有 Bug)
    → GET /orders/{id} (只读 DB, 不查 HeroSMS) ← BUG
    → 或用户手动点"刷新"
    → GET /orders/{id}/sms
    → backend: 调 heroSMS->getSms(hero_order_id)
    → 提取验证码, 返回给前端
    → 前端显示 SMS 码
```

## 六、总结

**好的方面**：
- 项目架构清晰，前后端职责分明
- 后端路由模块化设计良好
- 数据库事务使用正确，积分扣除/退款逻辑完善
- 前端 UI 完整美观，用户体验流畅
- 支持中英文国际化
- 订单生命周期管理完善（pending→active→completed/expired/cancelled）
- 价格系数系统灵活，支持按服务配置不同加价率
- 会员等级体系完善

**主要短板**：
1. **SMS 自动轮询有 Bug** — 轮询不触发 HeroSMS 查询，需手动刷新
2. **缺少 Webhook 回调** — 没有从 HeroSMS 接收 SMS 状态更新的机制
3. **安全风险** — 硬编码 API Key、密码重置无验证、user_id 明文传参
4. **支付局限** — 仅支持 Apple IAP，Web 端支付收据伪造
5. **前端价格显示不一致** — 硬编码乘数与后端系数不匹配

总体来看，产品核心链路**基本可走通**，但 SMS 自动接收和支付环节有明显缺陷，需要优先修复。
