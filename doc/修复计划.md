## 🔴 P0-1: 激活时未调用 setStatus(1) 告知 HeroSMS 开始等待短信

**影响**：即使 webhook 正确配置，HeroSMS 端可能不会触发回调。

**文件**：`backend/routes/orders.php` 第 150-228 行

**修复**：

```php
// 当前代码 (line 210-213):
$db->query(
    "UPDATE orders SET status = 'active', phone_number = ?, hero_order_id = ?, activated_at = NOW(), expires_at = DATE_ADD(NOW(), INTERVAL 20 MINUTE) WHERE id = ?",

// 修复后 — 加上 setStatus(1):
```

## 🔴 P0-2: Webhook URL 从未注册到 HeroSMS

**问题**：`HeroSMS` 类有完整的 `setWebhookUrl()` / `getWebhookUrl()` 方法，但系统中没有任何地方调用它们。`webhook.php` 实现完整但 HeroSMS 不会主动 POST 过来，因为 URL 未配置。

**影响**：短信到达后系统收不到通知，数据库不会更新，前端轮询只能读到旧数据。

**涉及文件**：
- `backend/webhook.php` — 已完整实现 ✅
- `backend/lib/HeroSMS.php` 第 421-461 行 — `setWebhookUrl()` 已实现 ✅
- **缺失**：初始化和注册 webhook 的脚本或启动逻辑 ❌

**修复方案A — 创建一次性初始化脚本**：

创建 `backend/setup_webhook.php`：

```php
<?php
/**
 * 配置 HeroSMS Webhook 地址（运行一次即可）
 * 
 * 使用: cd backend && php setup_webhook.php
 */
require_once __DIR__ . '/config/database.php';
require_once __DIR__ . '/lib/Database.php';
require_once __DIR__ . '/lib/HeroSMS.php';
require_once __DIR__ . '/lib/KeyManager.php';

// 从 .env 读取本服务的访问地址
$env_file = __DIR__ . '/.env';
if (!file_exists($env_file)) {
    die("错误: .env 文件不存在\n");
}
$env = parse_ini_file($env_file, false, INI_SCANNER_RAW);
$appUrl = rtrim($env['APP_URL'] ?? '', '/');

if (empty($appUrl)) {
    die("错误: APP_URL 未在 .env 中配置\n");
}

$webhookUrl = $appUrl . '/webhook/hero-sms';

echo "Webhook URL: $webhookUrl\n";
echo "继续配置? (y/n): ";
$confirm = trim(fgets(STDIN));
if (strtolower($confirm) !== 'y') {
    die("已取消\n");
}

$heroSmsApiKey = KeyManager::getHeroSmsApiKey();
if (empty($heroSmsApiKey)) {
    die("错误: HeroSMS API Key 未在数据库 system_settings 中配置\n");
}

$heroSMS = new HeroSMS($heroSmsApiKey, HEROSMS_BASE_URL);
$result = $heroSMS->setWebhookUrl($webhookUrl);

if ($result['success']) {
    echo "Webhook 配置成功!\n";
    echo "Webhook URL: $webhookUrl\n";
} else {
    echo "配置失败: " . ($result['message'] ?? $result['error'] ?? '未知错误') . "\n";
}
```

**修复方案B — 在系统启动时自动注册（推荐）**：

在后端 `index.php` 的初始化阶段加一段：

```php
// 自动注册 Webhook（仅在 URL 有变化时更新）
$cachedWebhookUrl = getSetting($db, 'webhook_configured_url', '');
$currentWebhookUrl = (defined('APP_URL') ? APP_URL : '') . '/webhook/hero-sms';

if ($cachedWebhookUrl !== $currentWebhookUrl && !empty($currentWebhookUrl)) {
    $heroSMS->setWebhookUrl($currentWebhookUrl);
    // 保存已配置的 URL，避免每次请求都重复设置
    $db->query(
        "INSERT INTO system_settings (`key`, `value`, `updated_at`) VALUES ('webhook_configured_url', ?, NOW())
         ON DUPLICATE KEY UPDATE `value` = ?, `updated_at` = NOW()",
        [$currentWebhookUrl, $currentWebhookUrl]
    );
}
```

## 🔴 P0-3: 前端 SMS 轮询缺少直接查询 HeroSMS 的备用路径

**问题**：前端 `_startSmsPolling()` 只调 `loadOrderDetail()`（读 DB）。在 webhook 已正确配置的情况下这是 OK 的，但如果 webhook 延迟或被 HeroSMS 跳过，用户永远收不到 SMS。

**影响**：即使修复了 P0-1 和 P0-2，仍缺乏直接查询作为兜底。

**文件**：`app_Flutter/lib/screens/orders/order_detail_screen.dart` 第 40-77 行

**修复**：

```dart
void _startSmsPolling() {
    _smsPollingTimer?.cancel();
    _isPolling = true;
    
    // 防重复标记
    bool _smsReceived = false;
    
    _smsPollingTimer = Timer.periodic(const Duration(seconds: 3), (timer) async {
      final order = context.read<OrderProvider>().currentOrder;
      if (order == null || !order.isActive || order.isCompleted || _smsReceived) {
        timer.cancel();
        _isPolling = false;
        return;
      }
      
      try {
        // 方式1：直接查询 HeroSMS（主动拉取，不依赖 webhook）
        final code = await context.read<OrderProvider>().getSmsCode(widget.orderId);
        
        if (code != null && code.isNotEmpty && mounted) {
          _smsReceived = true;
          timer.cancel();
          _isPolling = false;
          
          ScaffoldMessenger.of(context).showSnackBar(
            SnackBar(
              content: Text(context.loc.translate('sms_received')),
              backgroundColor: AppColors.success,
              action: SnackBarAction(
                label: context.loc.translate('copy'),
                textColor: Colors.white,
                onPressed: () {
                  Clipboard.setData(ClipboardData(text: code));
                },
              ),
            ),
          );
          return;
        }
        
        // 方式2：同时读取数据库（webhook 可能已更新）
        await context.read<OrderProvider>().loadOrderDetail(widget.orderId);
        final updatedOrder = context.read<OrderProvider>().currentOrder;
        
        if (updatedOrder != null && updatedOrder.smsCode != null && mounted) {
          _smsReceived = true;
          timer.cancel();
          _isPolling = false;
          
          ScaffoldMessenger.of(context).showSnackBar(
            SnackBar(
              content: Text(context.loc.translate('sms_received')),
              backgroundColor: AppColors.success,
              action: SnackBarAction(
                label: context.loc.translate('copy'),
                textColor: Colors.white,
                onPressed: () {
                  if (updatedOrder.smsCode != null) {
                    Clipboard.setData(ClipboardData(text: updatedOrder.smsCode!));
                  }
                },
              ),
            ),
          );
        }
      } catch (e) {
        // 轮询失败不中断，继续下一次尝试
        debugPrint('SMS polling error: $e');
      }
    });
  }
```

**同时修复 `OrderProvider.getSmsCode()` 使其不立即将本地状态改为 completed**：

文件：`app_Flutter/lib/providers/order_provider.dart` 第 251-311 行

当前代码在收到 SMS 后会把 `_currentOrder` 状态改为 `'completed'`，但后端订单状态仍是 `'active'`（需要 webhook 或用户再次查询才能知道）。修复方案：不修改本地状态，只返回 code。

```dart
// 修改 getSmsCode，删除以下代码块中设置 status='completed' 和 completedAt 的部分：
// 仅返回 code 字符串，不修改本地订单状态
// 本地订单状态由 loadOrderDetail（webhook 已更新 DB）来同步
```

## 🟠 P1-3: 前端硬编码 API Key

**问题**：`app_config.dart` 中 API Key 硬编码，反编译即可获取。

**影响**：可伪造客户端请求。

**文件**：`app_Flutter/lib/core/config/app_config.dart` 第 11-14 行

**修复方案（短期，降低风险）**：

```dart
// 将 API Key 存储为运行时变量，从启动时服务端下发
// 同时限制 API Key 仅对公开路径有效（后端已实现公开路径不需 API Key 检查）
// 最重要的是在后端加强 user_id 的校验（requireOrderOwner 等）
```

**后端加固**（优先级更高）：

在所有需要 user_id 的受保护端点，**优先使用 Token 认证**（从 Authorization Header 中解析 user_id），**禁止接受 query parameter 中的 user_id**，除非 token 解析失败才作为 fallback。

当前大部分端点的模式：
```php
$userId = $_GET['user_id'] ?? null;
if (!$userId) {
    $userId = getCurrentUserIdFromToken();
}
```

应改为：
```php
$userId = getCurrentUserIdFromToken(); // 优先 Token
if (!$userId) {
    // Token 认证失败才允许 query param（仅开发模式）
    if (APP_ENV === 'development') {
        $userId = $_GET['user_id'] ?? null;
    }
}
```

## 🟡 P2-3: 已激活订单超时无退款

**文件**：`backend/routes/orders.php` 第 267 行

**问题**：`cancelOrder` 只允许 pending 状态取消。如果号码激活后超时未收到 SMS，用户积分不退。

**修复**：
- 区分 "用户主动取消" 和 "系统超时取消"
- 系统超时取消（active 超 20min 未收 SMS）应该退积分（因为号码没收到验证码）
- 当前 cron 脚本直接把超时订单标为 completed 不退积分

cron 脚本修改 (`backend/cron/check-timeout.php`):
```php
// 当前：直接标记为 completed（不退分）
$db->query("UPDATE orders SET status = 'completed', completed_at = NOW() WHERE id = ?", [$orderId]);

// 改为：标记为 expired 并退款
$db->beginTransaction();
try {
    $db->query("UPDATE orders SET status = 'expired', updated_at = NOW() WHERE id = ?", [$orderId]);
    // 退款
    $refundAmount = intval($order['total_price']);
    $db->query("UPDATE users SET balance = balance + ? WHERE id = ?", [$refundAmount, $order['user_id']]);
    $balanceAfter = $db->query("SELECT balance FROM users WHERE id = ?", [$order['user_id']])->fetchColumn();
    $db->insert('credit_transactions', [
        'user_id' => $order['user_id'],
        'type' => 'refund',
        'amount' => $refundAmount,
        'balance_after' => $balanceAfter,
        'description' => "订单 #$orderId 超时自动退款"
    ]);
    $db->commit();
} catch (Exception $e) {
    $db->rollBack();
}
```
